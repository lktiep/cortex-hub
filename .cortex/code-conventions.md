## Naming Conventions

| Scope | Convention | Example |
|---|---|---|
| Variables & functions | `camelCase` | `getUserById`, `apiKeyHash` |
| Types & interfaces | `PascalCase` | `KnowledgeItem`, `QualityReport` |
| Constants | `UPPER_SNAKE_CASE` | `MAX_RATE_LIMIT`, `DEFAULT_TTL` |
| Files (components) | `PascalCase.tsx` | `MetricCard.tsx`, `DataTable.tsx` |
| Files (utilities) | `camelCase.ts` | `formatDate.ts`, `hashApiKey.ts` |
| Files (routes) | `kebab-case.ts` | `knowledge.ts`, `quality.ts` |
| CSS variables | `--kebab-case` | `--bg-primary`, `--accent` |
| Database columns | `snake_case` | `created_at`, `agent_id` |

## Import Style

Always use package path aliases — never relative paths across packages:

```typescript
// ✅ Correct
import type { KnowledgeItem } from '@cortex/shared-types'
import { formatDate } from '@cortex/shared-utils'
import { MetricCard } from '@cortex/ui-components'

// ❌ Incorrect
import type { KnowledgeItem } from '../../../packages/shared-types/src/models'
```

Within a package, relative imports are acceptable for same-directory files.

## File Organization

```
monorepo-packages pattern:
├── packages/     → shared code (types, utils, UI components)
├── apps/         → applications (hub-mcp, dashboard-api, dashboard-web)
└── infra/        → infrastructure (Docker, scripts, configs)
```

Each file has a single responsibility. Each package exports through `index.ts` barrel files.

## Error Handling

Use custom error classes that extend a base error:

```typescript
export class CortexError extends Error {
  constructor(
    message: string,
    public code: string,
    public statusCode: number = 500
  ) {
    super(message)
    this.name = 'CortexError'
  }
}

export class AuthError extends CortexError {
  constructor(message: string) {
    super(message, 'AUTH_ERROR', 401)
  }
}

export class PolicyViolationError extends CortexError {
  constructor(message: string, public policy: string) {
    super(message, 'POLICY_VIOLATION', 403)
  }
}
```

## API Pattern

Hono routes with clear separation of concerns:

```typescript
// routes/knowledge.ts — HTTP layer only
app.get('/knowledge', async (c) => {
  const result = await knowledgeService.search(c.req.query('q'))
  return c.json(result)
})

// services/KnowledgeService.ts — business logic
export class KnowledgeService {
  async search(query: string): Promise<KnowledgeItem[]> { /* ... */ }
}
```

## TypeScript Rules

- **Strict mode** enabled in all `tsconfig.json`
- **No `any`** without explicit `// eslint-disable-next-line` + comment explaining why
- **Prefer `type` over `interface`** for data shapes; `interface` only for contracts/services
- **Use `const` assertions** for literal types

## Testing Approach

- **Vitest** for unit and integration tests
- **Co-located tests**: `foo.ts` → `foo.test.ts` in the same directory
- **Test file naming**: `*.test.ts` (unit), `*.integration.test.ts` (integration)
- **Minimum coverage target**: 70% (enforced by quality gate)
